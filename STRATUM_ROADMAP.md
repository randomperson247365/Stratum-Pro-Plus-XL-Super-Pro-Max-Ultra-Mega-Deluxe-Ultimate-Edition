## Design Spec: Window Decorations

Inspired by KDE вҖ” SSD by default, drawn entirely by stratum-wm.

### Titlebar
- Height: 28px
- Font: UI font from config, medium weight
- Active: accent color background, white text
- Inactive: desaturated, ~60% opacity вҖ” clearly inactive, not ugly
- Corner radius: matches `border_radius` in config (default 8px)
- Radius only on top corners вҖ” bottom flush with window content

### Buttons (close / min / max)
- Right-aligned by default, configurable to left
- 16px SVG icons, 28px hit area
- Close: neutral inactive вҶ’ red on hover
- Min/Max: neutral, darken on hover
- No text labels, icons only

### Border
- Single clean border вҖ” NO triple-layer nonsense
- Width: 1px inactive / 2px active (configurable)
- Active: accent color
- Inactive: muted grey, barely there
- Same corner radius as titlebar

### Shadow
- Soft gaussian, active windows only
- Inactive windows: no shadow (helps depth perception)
- Spread: ~12px, opacity ~40% (configurable)

### Config
```toml
[decorations]
titlebar_height = 28
border_width_active = 2
border_width_inactive = 1
border_radius = 8
shadow_enabled = true
shadow_spread = 12
shadow_opacity = 0.4
buttons_position = "right"   # "left" | "right"
buttons = ["minimize", "maximize", "close"]
```

## Design Spec: Panel System

### Philosophy
Fully widget-based like KDE. Any number of panels, any screen edge,
any widget in any slot. Stored in config.toml, editable via panel
editor in stratum-settings with drag-and-drop.

### Panel Config
```toml
[[panels]]
id = "main-bar"
screen = 0           # 0 = primary, 1 = second monitor
position = "bottom"  # "top" | "bottom" | "left" | "right"
height = 40
autohide = false
opacity = 1.0

# Three zones: left, center, right
[[panels.left]]
type = "workspaces"

[[panels.left]]
type = "window_title"

[[panels.center]]
type = "clock"
format = "%H:%M"
show_date = true

[[panels.right]]
type = "media"

[[panels.right]]
type = "tray_focused"
pinned_app_id = "spotify"  # the ONE icon to always show

[[panels.right]]
type = "tray"              # full tray for everything else

[[panels.right]]
type = "battery"
show_percentage = true

[[panels.right]]
type = "network"

[[panels.right]]
type = "quick_settings"
```

### Focused Tray Widget
- Shows exactly ONE tray icon вҖ” the pinned app
- If that app isn't running, widget collapses to zero width (invisible, no gap)
- Click passes through to the app's normal tray handler
- Use case: always-visible Spotify / VPN / etc without full tray clutter

### Panel Editor (stratum-settings)
- Visual bar representation with left/center/right zones
- Drag widgets between zones, reorder within zones
- Click widget вҶ’ configure its options inline
- "Add widget" picker, "Add panel" button
- Writes directly to config.toml, live reloaded via inotify

## IPC Design

Unix socket at `$XDG_RUNTIME_DIR/stratum.sock`, JSON messages.

```
stratum-wm  <в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ>  stratum-shell
                  stratum.sock
stratum-settings в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ>  stratum-wm
```

Key messages:
- `ConfigChanged { key, value }` вҖ” broadcast on TOML change
- `FocusChanged { app_id, title }` вҖ” WM вҶ’ shell
- `WorkspaceChanged { index }` вҖ” WM вҶ’ shell
- `SpawnApp { command }` вҖ” launcher вҶ’ WM
- `ToggleLayout` вҖ” shell вҶ’ WM
- `OpenLauncher` вҖ” keybind вҶ’ shell
- `GetWindowList` вҖ” shell/settings вҶ’ WM

## Notes for LLM-assisted Development

- `stratum-config/src/schema.rs` is the single source of truth for all config.
- All structs use `#[serde(default)]` вҖ” partial configs always work.
- Panel widgets use an enum `WidgetType` вҖ” adding a widget means adding
  a variant, an Iced view impl, and a config struct. That's it.
- IPC uses dot-path keys matching TOML paths for `ConfigChanged` so
  components subscribe only to what they care about.
- `stratum-wm` is the ONLY binary speaking `river-window-management-v1`.
  Everything else uses IPC.
- Focused tray collapses to zero width when pinned app is not running вҖ”
  no empty space, no placeholder.
- The panel editor in settings writes config.toml directly. The shell
  picks up changes via inotify and rebuilds the panel live.

---

I'm building a Linux desktop environment called **Stratum Pro Plus XL Super Pro Max Ultra Mega Deluxe Ultimate Edition** (Stratum for short). This document is the full spec and roadmap вҖ” read it entirely before writing any code.

### The stack
- **Compositor:** River (https://codeberg.org/river/river) вҖ” a non-monolithic Wayland compositor. River handles rendering; our code handles window management via the `river-window-management-v1` protocol.
- **Language:** Rust throughout
- **UI framework:** Iced (for the shell and settings app)
- **Structure:** Cargo workspace with 6 crates (see workspace layout above)

### Where to start: Phase 1

Scaffold the full Cargo workspace first, then implement in this order:

1. **`stratum-config`** вҖ” the config schema crate. This is the foundation everything else depends on. Write `schema.rs` with full serde structs matching the config schema in this doc. All fields must have `#[serde(default)]`. Include `defaults.rs` and a basic `watcher.rs` using the `notify` crate for inotify-based live reload.

2. **`stratum-ipc`** вҖ” Unix socket IPC. Implement the message types from the IPC section and a basic async server/client using tokio.

3. **`stratum-wm`** вҖ” the window manager binary. This is the most complex piece:
   - Vendor `river-window-management-v1.xml` from the River repo into `protocols/`
   - Write `build.rs` to run `wayland-scanner` on it and generate Rust bindings
   - Implement a minimal event loop that connects to River as a Wayland client
   - Basic floating layout: windows open, can be moved and resized, can be closed
   - Keybinding dispatch reading from the config

4. **`contrib/river-init`** вҖ” a shell script that River runs on startup, which launches `stratum-wm`

### Reference
For how to wire up `river-window-management-v1` bindings in Rust, study the Canoe window manager source: https://github.com/roblillack/canoe вҖ” it's a stacking Rust WM for River and is the closest existing reference. Use `wayland-client` and `wayland-scanner` from the Smithay project.

### Rules
- Always run `cargo build` after each significant change and fix all errors before moving on
- Keep each crate focused вҖ” `stratum-wm` is the ONLY binary that speaks the River protocol
- All config structs live in `stratum-config`, imported by other crates
- Don't start Phase 2 (decorations) until Phase 1 fully compiles and connects to River
